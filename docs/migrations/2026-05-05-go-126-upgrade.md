# 2026-05-05: Go 1.26 化 (sqlc 最新版の要求への追従)

## Why

dependabot PR で `actions/setup-go v5 → v6` の CI が失敗。原因は **setup-go ではなく sqlc の最新版が Go 1.26 を要求** していたこと:

```
go: github.com/sqlc-dev/sqlc@v1.31.1 requires go >= 1.26.0
   (running go 1.25.9; GOTOOLCHAIN=local)
```

`ci.yml` で `go install sqlc@latest` していたため、sqlc が新マイナーをリリースするたびに CI が壊れる構造になっていた。同じ問題は **通常の master push でも近日中に表面化** する地雷だった。

## What

変更ファイル:
- `go.mod` (`go 1.25.x` → `go 1.26.0`)
- `.github/workflows/ci.yml` (`go-version: '1.25'` → `'1.26'`)

## How

### `go.mod`

```diff
- go 1.25.3
+ go 1.26.0
```

`go` ディレクティブは「最低限の Go バージョン要求」なので、ローカルが 1.26.x なら問題なし。`1.26.0` と書いて `1.26.5` で動かすのは正常。

### `.github/workflows/ci.yml`

```diff
- go-version: '1.25' # マイナーまで指定し、check-latest で常に最新パッチを取得
+ go-version: '1.26' # マイナーまで指定し、check-latest で常に最新パッチを取得
  check-latest: true
```

`check-latest: true` のおかげで、CI は 1.26 系の最新パッチを常に取得する。stdlib の脆弱性 (`crypto/tls`, `crypto/x509`, `net/mail` 等) も自動で取り込める。

## 設計判断: 固定 vs 追従

選択肢:
- **A**: ci.yml で `go install sqlc@v1.30.0` のようにバージョン固定 (短期しのぎ)
- **B**: Go バージョンを 1.26 に上げて sqlc 最新に追従 (採用)

A は次に sqlc がメジャー更新したら再対応が必要。B は sqlc を含む Go ツール群の最新を取り続ける方針と整合する。テンプレートの「常に最新パッチを使う」設計と一致するため B を採用。

## 派生プロジェクトへの適用

派生プロジェクトが Go 1.25 系で停止している場合、同じ問題が起きる前に上げておく:

派生プロジェクトの Claude Code に投げるプロンプト例:

```
テンプレリポの docs/migrations/2026-05-05-go-126-upgrade.md を参照して、
このプロジェクトの Go バージョンを 1.26 に上げてください。

ローカルの go バージョンが 1.26 以上であることを確認してから進めてください
(go version で確認)。
```

## 検証

- `go vet ./...` 緑
- `go test ./...` 緑
- `make lint` 緑
- CI で `setup-go` の最新版 + sqlc 最新版がインストールできる
